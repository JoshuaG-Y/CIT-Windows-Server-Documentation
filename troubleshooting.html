Troubleshooting Windows Server 2025

Common Issues and Solutions

General Troubleshooting Steps

  • Check Event Viewer: Windows Event Viewer (eventvwr.msc) is your first stop. Look under "Windows Logs" (Application, Security, System) and "Applications and Services Logs" for specific errors.
  • Review Log Files: Many applications (IIS, SQL Server) have their own detailed log files. Check their respective documentation for log locations.
  • Verify Network Connectivity: Use ping, tracert, and nslookup to diagnose network issues. Check firewall settings (Windows Defender Firewall).
  • Check Service Status: Ensure necessary services (IIS, SQL Server, SSHD) are running using services.msc or PowerShell's Get-Service.
  • Restart Services/Server: Sometimes a simple restart of a service or the entire server can resolve transient issues.
  • Check Resource Usage: Use Task Manager or Resource Monitor to check CPU, memory, disk, and network usage. High usage can indicate performance bottlenecks.
  • Ensure Latest Updates: Install the latest Windows Updates and updates for all installed software.
  • Consult Documentation: Refer to official Microsoft documentation and community forums (TechNet, Stack Overflow).

Specific Issue Troubleshooting

1. Windows Server 2025 Installation Issues

  • Problem: Installation fails or hangs.
    • Cause: Insufficient hardware resources (RAM, CPU, disk space), incompatible hardware, corrupted installation media, incorrect BIOS/UEFI settings.
    • Solution: Verify hardware meets minimum requirements. Check CIT hardware compatibility list. Re-download installation media or try a different USB/DVD. Ensure BIOS/UEFI is set to boot from the correct device and AHCI mode for SATA is enabled.
  • Problem: Drivers not found during or after installation.
    • Cause: Missing or incorrect drivers for specific CIT hardware components.
    • Solution: Download the latest drivers from the hardware manufacturer's website specifically for Windows Server 2025. Use Device Manager to identify missing drivers and update them.

2. Virtual IP and drhscit.org Configuration

  • Problem: Server not accessible via drhscit.org or its virtual IP.
    • Cause: Incorrect DNS configuration for drhscit.org. Incorrect network adapter settings on the server (IP, subnet, gateway, DNS). Firewall blocking traffic. Network routing issues.
    • Solution: Verify DNS A record for drhscit.org points to the correct virtual IP. Check server's IP configuration (ipconfig /all). Ensure Windows Firewall or any network firewall allows traffic on necessary ports (e.g., 80, 443). Work with network administrators to verify routing and NAT/PAT configurations for the virtual IP.

3. RDP and SSH Accessibility

  • Problem: Cannot connect via RDP.
    • Cause: RDP disabled on the server. Windows Firewall blocking RDP port (TCP 3389). Network connectivity issues. Incorrect credentials. Remote Desktop Services not running. Network Level Authentication (NLA) issues.
    • Solution: Ensure RDP is enabled (System Properties -> Remote). Check Windows Firewall rules for Remote Desktop. Verify network path using ping and tracert. Double-check username and password. Ensure "Remote Desktop Services" and "Remote Desktop Services UserMode Port Redirector" services are running. If NLA is an issue, try connecting from a domain-joined machine or temporarily disable NLA for testing (not recommended for production).
  • Problem: Cannot connect via SSH.
    • Cause: OpenSSH Server feature not installed or configured. SSH service (sshd) not running. Windows Firewall blocking SSH port (TCP 22). Incorrect SSH key configuration or password.
    • Solution: Install OpenSSH Server (Settings -> Apps -> Optional features). Start the sshd service (Start-Service sshd, Set-Service sshd -StartupType Automatic). Add a firewall rule for port 22 (New-NetFirewallRule -Name sshd -DisplayName 'OpenSSH Server (sshd)' -Enabled True -Direction Inbound -Protocol TCP -Action Allow -LocalPort 22). Verify SSH key paths and permissions in %programdata%\ssh\sshd_config if using key-based authentication. Check sshd logs in Event Viewer (Applications and Services Logs -> OpenSSH).

4. IIS and SQL Server (Developer Edition) Issues

  • Problem: Web application on IIS not accessible or shows errors (e.g., HTTP 500, 403, 404).
    • Cause: IIS not installed or configured correctly. Application pool identity permissions issues. Incorrect web.config settings. Missing application dependencies. Firewall blocking HTTP/HTTPS ports. DNS issues.
    • Solution: Verify IIS role and required features are installed. Check application pool identity has necessary permissions to application files and database. Review IIS logs (%SystemDrive%\inetpub\logs\LogFiles). Enable detailed error messages in IIS for easier debugging. Ensure bindings are correct (IP address, port, hostname). Check Windows Firewall.
  • Problem: Cannot connect to SQL Server Developer Edition.
    • Cause: SQL Server service not running. SQL Server not configured for remote connections. TCP/IP protocol disabled in SQL Server Configuration Manager. Windows Firewall blocking SQL Server port (default TCP 1433). Incorrect connection string. SQL Server Browser service not running (if using named instances).
    • Solution: Ensure "SQL Server (MSSQLSERVER)" service is running. Use SQL Server Configuration Manager to enable TCP/IP protocol and check the port. Add a firewall rule for the SQL Server port. Verify connection string details (server name, database, authentication). Ensure SQL Server Browser service is running if connecting to a named instance. Check SQL Server error logs.
  • Problem: Full-stack application database connectivity issues.
    • Cause: Incorrect database connection string in the application. SQL Server user permissions insufficient. Network issues between web server and database server (if separate).
    • Solution: Double-check the connection string. Ensure the SQL login used by the application has appropriate permissions (db_datareader, db_datawriter, etc.) on the target database. Test connectivity from the web server to the SQL server using SQL Server Management Studio (SSMS) or a UDL file.

5. KeePass Credential Manager Issues

  • Problem: KeePass fails to install or run.
    • Cause: Corrupted download. .NET Framework issues (KeePass 2.x requires .NET). Insufficient user permissions for installation.
    • Solution: Re-download KeePass from the official website. Ensure the required .NET Framework version is installed and updated. Run the installer as an administrator.
  • Problem: Cannot open KeePass database.
    • Cause: Incorrect master password or key file. Corrupted database file. File permissions issues.
    • Solution: Carefully verify the master password and key file location/integrity. Try opening a backup of the database if available. Ensure you have read/write permissions to the database file and its folder.

6. Recommended Software Issues

  • Problem: PowerShell 7 installation or execution issues.
    • Cause: Incorrect architecture downloaded (x64 vs Arm64). Execution policy restrictions.
    • Solution: Download the correct installer for your server's architecture. Check PowerShell execution policy (Get-ExecutionPolicy) and set it appropriately if needed (e.g., Set-ExecutionPolicy RemoteSigned).
  • Problem: Windows Admin Center connectivity issues.
    • Cause: WAC gateway not running or misconfigured. Firewall blocking WAC ports. WinRM not configured correctly on managed servers. TrustedHosts issues.
    • Solution: Ensure WAC service is running. Check firewall rules on both WAC gateway and managed server. Configure WinRM on managed servers (winrm quickconfig). Add managed server FQDNs or IPs to TrustedHosts on the WAC gateway if not domain-joined (Set-Item WSMan:\localhost\Client\TrustedHosts -Value 'server.fqdn.com' -Force).
  • Problem: Git for Windows command line or GUI issues.
    • Cause: Incorrect installation options (e.g., PATH integration). SSH key issues for remote repositories. Proxy configuration.
    • Solution: Reinstall Git, paying attention to PATH setup. Configure SSH keys correctly for services like GitHub/GitLab. Configure Git proxy settings if behind a corporate proxy (git config --global http.proxy http://proxyuser:proxypwd@proxy.server.com:8080).

7. Security Best Practices Implementation

  • Problem: Difficulty applying specific security configurations.
    • Cause: Lack of understanding of Group Policy Objects (GPOs), Local Security Policy, or specific security tool configurations. Conflicts with existing settings.
    • Solution: Consult Microsoft's security baselines for Windows Server 2025. Use tools like Security Compliance Manager (SCM) or Microsoft Security Configuration Toolkit. Test changes in a non-production environment first. Use gpresult /h report.html to check applied GPOs.
  • Problem: Windows Defender Antivirus or Firewall blocking legitimate applications.
    • Cause: Overly restrictive rules or false positives.
    • Solution: Create specific exclusions for application executables or folders in Windows Defender Antivirus. Create specific inbound/outbound rules in Windows Defender Firewall for necessary application ports and protocols.

8. Documentation Publishing to drhscit.org (GitHub Pages)

  • Problem: Changes not appearing on the GitHub Pages site.
    • Cause: Git changes not committed and pushed to the correct branch (e.g., main, gh-pages). GitHub Actions build/deployment failure. Caching issues in the browser or CDN. Incorrect GitHub Pages settings in the repository.
    • Solution: Ensure all local changes are committed (git add ., git commit -m "message") and pushed (git push origin ). Check the "Actions" tab in your GitHub repository for build logs. Clear browser cache or try incognito mode. Verify GitHub Pages source branch and folder settings in repository settings.
  • Problem: Static site generator (if used) errors.
    • Cause: Incorrect configuration files for the generator (e.g., Jekyll, Hugo). Syntax errors in content files. Missing dependencies.
    • Solution: Check the static site generator's documentation for common errors. Validate configuration files. Build the site locally to identify errors before pushing to GitHub.
